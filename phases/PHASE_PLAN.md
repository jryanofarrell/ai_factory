# `ai_factory` Phase Plan

This is the master phase plan for `ai_factory`. It supersedes any earlier phase ordering. Each phase has a one-paragraph description and acceptance criteria. The stopping point for v1 is **Phase 5**.

## Operating constraints (v1)

- **Local execution only.** No GitHub Actions runner, no remote workers. The user runs `factory run` from their own machine.
- **Single model: Claude Sonnet.** Both ideation and execution use Sonnet via Claude Code. No model-routing logic in v1.
- **Single human reviewer.** The user reviews every PR. No auto-merge.

These constraints are also encoded in `docs/DECISIONS.md` (see the note at the end of this doc for the small edits to your saved Phase 0 spec).

---

## Phase 0 — Contracts and scaffolding

Set up the repo skeleton plus all design docs, memory, skills, and rules scaffolding that downstream phases depend on. No executable behavior beyond a CLI that prints help.

**Acceptance criteria**
- `uv sync` succeeds; `uv run factory --help` and `factory version` both work.
- `CLAUDE.md`, `.claude/skills/README.md`, and all five `docs/*.md` files exist with real prose, not stubs.
- `docs/DECISIONS.md` contains nine ADRs (the original eight plus ADR-009 for local-only execution) and the pitfalls section.
- `examples/tickets/hello-world.md` conforms to `TICKET_SPEC.md`.
- `manifest.example.yaml` conforms to `MANIFEST.md`; `manifest.yaml` is in `.gitignore`.
- No code beyond the Typer scaffold.

---

## Phase 1 — Single-shot executor

`factory run-ticket <path-to-ticket.md> --repo <name>` takes a hand-written ticket file, invokes Claude Code (Sonnet) in headless mode against the target repo, runs tests, opens a PR via `gh`, and prints the URL. Linear is not involved.

**Acceptance criteria**
- The hello-world ticket runs end-to-end and produces a PR on `toms-hms/thms-platform`.
- Branch name follows `factory/<ticket-id>-<uuid>` convention.
- PR body includes the ticket's acceptance criteria and a "Generated by ai_factory" footer.
- Agent producing no changes → branch deleted, informative error printed.
- Test failure → branch preserved, no commit/push, error printed with branch name.
- Every external command (`git`, `gh`, `claude`) is echoed before running and its output streams live.
- Re-running the same ticket produces a second PR with a different uuid.
- `manifest.py` and `ticket.py` have unit tests covering happy path and common validation failures.

---

## Phase 2 — Linear read path

`factory pull-tickets` queries Linear for issues in the "Ready for Agent" workflow state (or carrying the `factory:ready` label), translates each into the on-disk ticket format, and writes them into a local queue directory. No execution yet — this phase only validates that we can structurally pull tickets out of Linear and that the schema works in practice.

**Acceptance criteria**
- `factory pull-tickets` writes one valid ticket file per ready Linear issue, conforming to `TICKET_SPEC.md`.
- Ticket frontmatter is populated from Linear custom fields (`target_repo`, `scope_paths`, `budget_tokens`, `budget_minutes`) and the issue description (acceptance criteria).
- Tickets that fail validation are reported with a clear, actionable message and are not written to disk.
- Team→repo resolution uses Linear's native GitHub integration metadata; if unavailable, falls back to a documented convention (team key matches repo name).
- Re-running `pull-tickets` is idempotent: same Linear state produces the same files, no duplicates.
- Linear API key is read from `.env`, never hardcoded; missing key produces a clear error.
- A unit test covers the Linear-issue → ticket-file translation against a fixture payload.

---

## Phase 3 — Closed loop

`factory run` composes Phases 1 and 2: pull ready tickets, execute each via the run-ticket pipeline, then write back to Linear (comment with PR link, move issue to "In Review", post cost and duration). Manual trigger; tickets processed sequentially.

**Acceptance criteria**
- `factory run` on a workspace with N ready tickets produces up to N PRs and N updated Linear issues.
- Each Linear issue gets a comment with the PR URL, the run cost, and the wall-clock duration.
- Successful runs move the Linear issue to "In Review" (or equivalent state); failures move it to "Failed" with an explanatory comment.
- Failures in one ticket do not block remaining tickets; each runs independently.
- A summary report is printed at the end: X succeeded, Y failed, total cost, total duration.
- The run is resumable: if interrupted mid-batch, the next invocation skips already-processed tickets in this run.

---

## Phase 4 — Hardening

Add the safety rails that make unattended (or even attended overnight) running trustworthy: scope checks, budget caps, idempotency, structured logs, dry-run mode.

**Acceptance criteria**
- A ticket with `scope_paths` set fails the run if the agent's diff touches files outside the allowlist; failure is reported to Linear with the offending paths.
- A ticket that exceeds `budget_tokens` or `budget_minutes` is killed mid-run; branch is deleted; Linear issue is moved to "Failed" with cost and time used.
- Stale `factory/*` branches older than a configurable threshold (default 7 days) are auto-deleted at the start of `factory run`.
- Every ticket run produces a structured JSON log at `logs/<date>/<ticket-id>.json` capturing cost, duration, files changed, exit code, and any error.
- A `--dry-run` flag runs the full pipeline up to (but not including) `git push`, leaving the branch and diff inspectable.
- A pre-commit secret scan on the agent's branch fails the run if any plausible secret is detected.

## Phase 5 — Ideation (stop here for v1)

`factory ideate` takes a brain dump (file path or stdin) and an optional `--repo` flag, calls Sonnet to draft a structured Linear issue with the right custom fields filled in, and creates the issue in Linear in a `Draft` (or `Backlog`) state. The human reviews and manually promotes to `Ready for Agent`.

**Acceptance criteria**
- `factory ideate <brain-dump.md> --repo thms-platform` creates a Linear issue in the correct team's Draft/Backlog state.
- The created issue has `acceptance_criteria`, `scope_paths`, `budget_tokens`, and `budget_minutes` populated (with sensible defaults for budgets when not derivable).
- Issue title and description are generated from the brain dump and reference the brain-dump file path in the description.
- The issue is never created in the "Ready for Agent" state. Promotion is always a manual human action in Linear.
- The command refuses to create a ticket if it cannot unambiguously resolve a target repo (either via `--repo` or from explicit mention in the brain dump).
- Cost and a link to the created issue are printed on completion.

After this phase you have the system you described: ideate creates a Linear ticket, you mark it ready, `factory run` executes ready tickets, PRs appear for review.

---

## Phase 6 — Local scheduling (optional, deferred)

A launchd plist (macOS) or cron job that invokes `factory run` on a schedule. Desktop notifications surface failures.

**Acceptance criteria**
- A documented setup procedure with an example launchd plist (and a cron fallback for Linux).
- Scheduled-run failures produce a desktop notification visible without opening a terminal.
- Scheduled runs append to `logs/scheduled/<timestamp>.log`.
- Manual `factory run` still works and is unaffected by the scheduling layer.

Defer until you actually want overnight runs. Until then, run `factory run` before bed.

---

## Phase 7 — Multi-repo (optional, deferred)

Add a second repo to the manifest and prove no logic is hardcoded to `thms-platform`. Mostly a refactor pass to find and remove single-repo assumptions.

**Acceptance criteria**
- Adding a new entry to `manifest.yaml` is the only configuration step required to onboard a second repo.
- `factory run` correctly routes tickets to the right repo based on Linear team→repo mapping.
- Each repo's `CLAUDE.md` is loaded by Claude Code only when that repo is the target.
- Tests pass independently for both repos using their respective `test_command` from the manifest.

Defer until you actually have a second repo you want the factory working on.

---

## Where to stop

**Phase 5.** After Phase 5 the system does end-to-end what you described: ideate → ticket → mark ready → run → PRs. Phases 6 and 7 are quality-of-life additions (automatic scheduling, second repo) that are easy to bolt on later when you actually need them. Building them speculatively risks shaping the core for use cases that may never materialize.

A reasonable cadence: knock out Phase 0 in one session, Phase 1 in another, then sit on the Phase 1 system for a few days using hand-written tickets to stress-test the executor before moving to Phase 2. Don't rush from 1 to 5 without using each level.

## Edits to make to your saved Phase 0 spec

The Phase 0 spec from earlier in this conversation was written before you settled on local-only and Sonnet-only. Two small changes keep `DECISIONS.md` accurate:

**Replace the ADR-002 entry** with this version:

> **ADR-002: Single model (Claude Sonnet) for both ideation and execution in v1.** Context: an earlier sketch considered cheap-on-ideation, strong-on-execution model routing, but for a personal factory the operational complexity outweighs the savings; coding execution especially benefits from a capable model. Decision: use Claude Sonnet for both ideation and execution in v1; do not build model-routing logic. Consequences: simpler implementation; higher per-ticket cost than a routing strategy could achieve; revisit only if specific repetitive ticket patterns make a cheaper executor obviously worth it.

**Append a new ADR-009** after ADR-008:

> **ADR-009: Local-only execution in v1.** Context: an earlier sketch considered GitHub Actions on cron for an unattended overnight runner, requiring secrets management and remote execution. Decision: v1 runs entirely on the user's local machine; `factory run` is invoked manually or via local scheduling (launchd/cron) in a deferred Phase 6. Consequences: no remote secrets to manage; auth is whatever the local user has set up (`gh auth`, `claude` login, Linear API key in `.env`); the local machine must be running for scheduled invocations; remote execution can be added later as an alternative path if needed.

Also update the `docs/PHASES.md` content section in the spec to reflect the revised phase ordering (Phase 5 is now ideation; Phase 6 is local scheduling; Phase 7 is multi-repo; both 6 and 7 are marked optional/deferred).

And update the acceptance criterion that mentions "all eight ADRs (ADR-001 through ADR-008)" to read "all nine ADRs (ADR-001 through ADR-009)".
