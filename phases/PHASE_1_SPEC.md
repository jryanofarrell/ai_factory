# Phase 1 — Single-Shot Executor

## Brief for Claude Code

You are building Phase 1 of `ai_factory`. Phase 0 produced design docs and a CLI scaffold. Phase 1 produces the first executable behavior: given a ticket file and a registered repo, the factory invokes Claude Code (Sonnet) in headless mode to do the coding work, runs tests, opens a GitHub PR, and prints the URL.

The factory runs entirely locally. No remote runner. Sonnet is the only model.

**Before starting, read these existing files in the repo, in this order:**
- `CLAUDE.md` (project memory; loaded automatically by Claude Code, but read it explicitly so you internalize the Rules section)
- `docs/DECISIONS.md` (architectural decisions and the reasoning behind them — do not relitigate these)
- `docs/ARCHITECTURE.md`
- `docs/MANIFEST.md`
- `docs/TICKET_SPEC.md`
- `docs/PHASES.md`

If anything is ambiguous for the work below, fix the doc before writing code. If you find yourself wanting to revise a decision in `DECISIONS.md`, stop and surface the disagreement rather than editing the existing decision in place.

**Linear is not involved in Phase 1.** Only the on-disk ticket file matters. The first end-to-end test target is `toms-hms/thms-platform`, which `jryanofarrell` already has push access to.

## Goal

After Phase 1, this command works end-to-end:

```bash
uv run factory run-ticket examples/tickets/hello-world.md --repo thms-platform
```

It produces a PR on `toms-hms/thms-platform` and prints the PR URL on stdout.

## Behavior — the pipeline, step by step

The `run-ticket` command must do the following, in order:

1. **Load `manifest.yaml`** at the repo root. If missing, error with a pointer to `manifest.example.yaml`.
2. **Resolve the target repo** — look up `--repo` in the manifest's `repos` map. Error clearly if not found. Expand `~` in `local_path`.
3. **Parse the ticket** — load the markdown file, parse YAML frontmatter, extract the body. Validate required fields (`id`, `title`, `target_repo`) and require an `## Acceptance Criteria` section in the body. Cross-check that the ticket's `target_repo` matches the `--repo` flag; warn on mismatch and let `--repo` win.
4. **Sync the local checkout**:
   - If `local_path` does not exist, `git clone` from `github` (the `owner/repo` field in the manifest entry) into that path.
   - Otherwise: `git fetch origin`, `git checkout <default_branch>`, `git pull --ff-only`.
   - If the working tree is dirty before this step, abort with a clear error.
5. **Create a branch** named `factory/<ticket-id-lower>-<short-uuid>` (use `uuid.uuid4().hex[:8]`). The uuid suffix prevents collisions on re-runs of the same ticket. Check it out.
6. **Invoke Claude Code in headless mode**. Construct a prompt that includes:
   - The full ticket content (frontmatter and body).
   - An instruction to make the requested changes in the current working directory.
   - An explicit reminder: do not run `git commit`, do not run `git push`, only edit files.

   Run `claude -p "<prompt>"` as a subprocess with `cwd=local_path`. Stream its stdout/stderr to the user's terminal in real time so progress is visible. Capture the exit code; non-zero exits abort the rest of the pipeline.

   **Do not** read the target repo's `CLAUDE.md` and inline it into the prompt. Claude Code auto-loads `CLAUDE.md` from `cwd`, so passing `cwd=local_path` is sufficient and avoids double-loading. This is how the factory respects each target repo's own memory and rules.

7. **Verify the agent produced changes** — run `git status --porcelain`. If empty, abort with a clear "agent produced no changes" error and clean up the branch (`git checkout <default_branch>`, `git branch -D <branch>`).
8. **Run tests** — execute the manifest's `test_command` in `local_path`. If tests fail, leave the branch in place (the user may want to inspect), print the test output and the branch name, and abort before pushing.
9. **Commit** — `git add -A`, then commit with message `<ticket-id>: <ticket-title>` and a body containing the relative ticket file path.
10. **Push** — `git push -u origin <branch>`.
11. **Open a PR** via the `gh` CLI:

    ```bash
    gh pr create \
      --title "<ticket-id>: <ticket-title>" \
      --body "<body>" \
      --base <default_branch> \
      --head <branch>
    ```

    The PR body should contain the ticket's acceptance criteria and a "Generated by ai_factory" footer.
12. **Print the PR URL** on stdout. Done.

## Implementation notes

- Use `subprocess.run` (no `shell=True`) for all external commands. For commands whose output you want to stream, use `subprocess.Popen` with `stdout=None, stderr=None` so it inherits the parent's TTY.
- Print every external command before running it, prefixed with `$ `, so the user can see what's happening.
- Use `pyyaml` for both manifest and ticket frontmatter. Parse frontmatter manually by splitting on `---` lines at the top of the file — don't add a `python-frontmatter` dependency for this.
- Module layout under `src/factory/`:
  - `cli.py` — Typer wiring; depends on the rest.
  - `manifest.py` — `load_manifest()`, `RepoConfig` dataclass.
  - `ticket.py` — `parse_ticket(path) -> Ticket` dataclass; validation.
  - `git_ops.py` — thin wrappers over `git`, `gh`, returning typed results.
  - `runner.py` — the orchestrating pipeline above.
- Each module gets basic pytest coverage for parsing/validation. The pipeline itself is integration-only — don't try to mock it in Phase 1.

## Auth and prerequisites (must be documented in README)

The factory does not manage credentials. The user is expected to have these set up before running:

- `git` installed and configured.
- `gh` CLI installed and authenticated (`gh auth status` passes) to a GitHub account with push access to the target repo.
- `claude` CLI installed and authenticated. The factory uses whatever model `claude` is configured with; the user should set this to Sonnet.

If `gh` or `claude` is not on PATH, the runner must error early with a useful message.

## Error handling

The pipeline must be safe to abort at any step:

- Agent produced no changes → branch is deleted before exit; error printed.
- Tests fail → branch is preserved; test output and branch name printed; no commit/push.
- Push fails → no PR is attempted; error printed; commit and branch preserved.
- PR creation fails → branch and commit are preserved; error printed with retry instructions.
- Any uncaught exception → print the traceback and the current branch name so the user can clean up manually.

## Acceptance criteria

- [ ] `uv run factory run-ticket examples/tickets/hello-world.md --repo thms-platform` produces a PR on `toms-hms/thms-platform` end-to-end.
- [ ] The PR's branch name matches `factory/thms-XX-<uuid>`.
- [ ] The PR body contains the ticket's acceptance criteria and a "Generated by ai_factory" footer.
- [ ] If the agent produces no changes, the branch is cleaned up and a useful error is printed.
- [ ] If tests fail, no PR is created; the branch is preserved with a printout.
- [ ] Every external command (git, gh, claude) is echoed to the terminal before running and its output streams live.
- [ ] `manifest.py` and `ticket.py` have unit tests covering happy path and the most likely validation failures.
- [ ] Running the same ticket twice produces two distinct PRs (uuid suffix in the branch).

## Out of scope — do not build

- Linear integration of any kind (Phase 2+).
- Scope/file allowlist enforcement based on `scope_paths` (Phase 4).
- Budget caps in tokens or wall time (Phase 4).
- Idempotency or cleanup of stale `factory/*` branches (Phase 4).
- Multi-ticket batching or concurrency (Phase 3).
- Local scheduling / cron (Phase 6, deferred).
- Any web UI or status dashboard.
- Skipping tests via a flag — tests must always run.

## Verification (manual, after Claude Code finishes)

1. `uv sync && uv run factory --help` — confirm `run-ticket` appears in the command list.
2. Copy `manifest.example.yaml` to `manifest.yaml` and adjust `local_path` to a real location on disk (e.g. `~/factory/repos/thms-platform`). Make sure the parent directory exists.
3. Run `uv run factory run-ticket examples/tickets/hello-world.md --repo thms-platform`. Watch the terminal stream the work.
4. Open the printed PR URL. Confirm the diff is sensible and matches the ticket's acceptance criteria.
5. Re-run the same command. Confirm a second PR opens with a different branch name and a different uuid suffix.
6. Manually edit `examples/tickets/hello-world.md` to remove the `## Acceptance Criteria` section and re-run. Confirm a clear validation error is printed and nothing else happens.
